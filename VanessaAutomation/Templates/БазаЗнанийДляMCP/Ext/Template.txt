Вопрос: С помощью каких шагов можно нажать на кнопку формы клиента тестирования?
Ответ: если кнопка находится в шапке формы, то на неё можно нажать с помощью шагов:
```Gherkin
И я нажимаю на кнопку с именем 'ИмяКнопки'
```
Если поиск кнопки будет происходить по имени кнопки.
```Gherkin
И я нажимаю на кнопку 'Заголовок кнопки'
```
Если поиск будет происходить по заголовку кнопки.
Важный момент: если на форме есть несколько кнопок с одинаковым заголовком, то при использовании шага
И я нажимаю на кнопку 'Заголовок кнопки'
могут возникнуть проблемы, т.к. не получится однозначно найти кнопку.
Поэтому в общем случае лучше использовать шаг:
И я нажимаю на кнопку с именем 'ИмяКнопки'

Если же кнопка принадлежит таблице формы, то нужно использовать шаги:
```Gherkin
И в таблице "ИмяТаблицы" я нажимаю на кнопку с именем 'ИмяКнопки'
```
Если поиск кнопки будет происходить по имени кнопки.
```Gherkin
И в таблице "ИмяТаблицы" я нажимаю на кнопку "Добавить"
```
Если поиск будет происходить по заголовку кнопки.
Такие кнопки также лучше искать по наименованию.
Чтобы найти все шаги, которые могут нажимать на кнопки надо выполнить поиск шагов инструментом search_for_steps_by_keywords с отбором search_type="UI.Таблицы|UI.Формы.Кнопки" и search_name="нажимаю на кнопку"

Вопрос: Как проверить какое-либо значение не на полное равенство, а на частичное совпадение?
Ответ: Для этого есть шаги, которые умеют работать по шаблону.
В конце такого шага есть специальная строка: "по шаблону". Тогда в параметре шага можно указывать символы *, чтобы выполнять частичное сравнение
Допустим в таблице поле "Товар" и оно имеет значение "Стулья"
Например, такой шаг делает строгую проверку значения поля:
```Gherkin
И в таблице "Товары" поле "Товар" имеет значение "Стулья"
```
а такой частичную проверку
```Gherkin
И в таблице "Товары" поле "Товар" имеет значение "Сту*" по шаблону
```
Чтобы получить все шаги, которые умеют делать частичное совпадение с эталоном надо выполнить поиск шагов с параметром 'search_name=по шаблону'

Вопрос: Как проверить текст сообщений пользователю?
Ответ: Чаще всего достаточно использовать шаг
```Gherkin
Тогда в логе сообщений TestClient есть строки:
    | 'Сообщение1' |
    | 'Сообщение2' |
    | 'СообщениеN' |
```

Однако может возникнуть ситуация, когда часть сообщения пользователя при проверке нужно игнорировать, например, если там содержится текущая дата.
Тогда следует использовать шаг:
```Gherkin
Тогда в логе сообщений TestClient есть строки по шаблону:
    | 'Сообщение1' |
    | 'Сообщение2' |
    | 'СообщениеN' |
```
В этом случае в строках Сообщение1..СообщениеN можно использовать символ *, который заменяет произвольное число символов. Символ * можно использовать много раз в одной строке.
Чтобы найти все шаги, которые могут проверять сообщения надо выполнить поиск шагов инструментом search_for_steps_by_keywords с отбором search_type="UI.Сообщения пользователю"

Вопрос: Какой синтаксис языка Turbo Gherkin?
Ответ: Язык Turbo Gherkin создан на основе языка Gherkin.
# В отличие от классического Gherkin, Turbo Gherkin поддерживает:
- Вызов подсценариев
- Использование условий (`Если...Тогда...ИначеЕсли...Иначе`)
- Использование циклов
- Переменные.
  - Есть возможность сохранить произвольное значение в переменную во время выполнения шага
  - Есть специальная секция переменных
- Группы шагов
- Конструкцию `Попытка/Исключение`
- Теги для управления выполнением

Основные конструкции языка Turbo Gherkin.
# Основные ключевые слова всего feature файла
  - Функциональность / Функционал - описание общей функциональности.
  - Контекст - шаги, которые выполняются перед каждым сценарием.
  - Сценарий - конкретный тест-кейс.
  - Структура сценария - шаблон для тестов с разными данными.
  - Примеры - таблица данных для структуры сценария.
  
## Структура feature-файла

### Директива языка

В самом начале файла может быть указана директива языка:

```Gherkin
# language: ru
```

Если эта строка не указана, язык по умолчанию считается русским.

### Теги фичи

После директивы языка идёт секция тегов. Теги — это метки, которые позволяют управлять выполнением фича-файлов:

```Gherkin
@lessons
@ExportScenarios
@recordvideo
@screenshot
```

**Назначение тегов:**
- `@ExportScenarios` — разрешает вызов сценариев из других файлов

### Заголовок функциональности

После тегов указывается название функциональности:

```Gherkin
Функционал: Создание заказа клиентам
```

или

```Gherkin
Функционал: Проверка остатков на складе
```

**Ключевые слова для заголовка:**
- `Функционал:`
- `Функциональность:`

Секция является **обязательной** и может встречаться только один раз в файле.

### Роль и бизнес-эффект (опционально)

После заголовка может идти секция с описанием роли пользователя и цели:

```Gherkin
Как менеджер по продажам
Я хочу иметь возможность отражать в системе заказы клиентов
Чтобы заключать сделки с клиентами
```

Эта секция является **необязательной**.

## Секции файла

### Секция "Контекст"

Секция `Контекст:` содержит шаги, которые выполняются **перед каждым сценарием** фича-файла:

```Gherkin
Контекст:
    Дано я подключаю библиотеку шагов
    И я закрываю все окна клиента тестирования
```

**Особенности:**
- Секция является необязательной
- Если секция объявлена, в ней должен быть хотя бы один шаг
- Если в файле есть Контекст, но нет сценариев, шаги Контекста не выполняются


### Секция "Структура сценария". Нужно использовать только при необходимости.

Структура сценария позволяет запускать один сценарий несколько раз с разными параметрами:

```Gherkin
Структура сценария: Структура сценария 1
    Дано я открываю форму с параметром "<Параметр1>"
    Когда я ввожу значение "<Параметр2>"
    Тогда я получаю результат "<Параметр3>"
    
    Примеры:
    | Параметр1 | Параметр2 | Параметр3 |
    | Значение1 | Значение2 | Значение3 |
    | Значение4 | Значение5 | Значение6 |
```

**Особенности:**
- Параметры указываются в угловых скобках `<Параметр>`
- После шагов следует обязательное ключевое слово `Примеры:`
- В таблице примеров первая строка — имена параметров
- Последующие строки — значения для каждого прогона

### Секция "Сценария"

Сценарии — это основные исполняемые блоки:

```Gherkin
@тегСценария
Сценарий: Имя_сценария
    Дано шаг 1
    Когда шаг 2
    Тогда шаг 3
```

**Правила:**
- Имя сценария должно быть уникальным в пределах файла
- У сценария могут быть свои теги
- **Важно:** не отделяйте номер сценария пробелом от названия
  - ❌ Неправильно: `Сценарий 1`, `Сценарий 2` (конфликт имён)
  - ✅ Правильно: `Сценарий_1`, `Сценарий_2`

## Шаги сценариев

### Ключевые слова шагов сценария
Каждый шаг сценария должен начинаться с какого-то ключевого слова.
В русском языке это будут:
  - Допустим / Пусть / Дано - начальные условия
  - Когда - действие пользователя.
  - Тогда / Затем / То - ожидаемый результат.
  - И / К тому же/ Также - дополнение к предыдущему шагу.
  - Но / А / Иначе - логическое отрицание или исключение.
Ключевое слово не несёт в себе смысловой нагрузки.
Если в шаге поменять одно ключевое слово на другое, то шаг будет выполнять ровно те же действия, что и до этого.

### Параметры шагов

Шаги могут принимать параметры трех типов:

| Тип | Синтаксис | Пример |
|-----|-----------|--------|
| Строка | В кавычках или апострофах | `"текст"` или `'текст'` |
| Число | Без обрамления | 10, 0, -5 |
| Дата | В формате ДД.ММ.ГГГГ | `08.04.1981`, `08.04.81` |

**Рекомендация:** Используйте апострофы для строк — у них выше приоритет.
**Рекомендация:** Лучше использовать строковые параметры

### Снипеты шагов

**Снипет** — это служебное имя шага, по которому Vanessa Automation находит соответствующую процедуру в библиотеке шагов.

**Алгоритм вычисления снипета:**
1. Отбрасывается ключевое слово в начале строки
2. Удаляются все параметры из шага
3. Оставшийся текст — это снипет

**Пример:**
```Gherkin
Дано ПримерШагаСПервымПараметромИВторымПараметром "Параметр1" "Параметр2"
```
Снипет: `ПримерШагаСПервымПараметромИВторымПараметром`

Если для снипета не найдена соответствующая процедура, будет выдано сообщение:
> Не найдена процедура или функция для выполнения шага

### Группы шагов

Группы шагов позволяют логически структурировать сценарий:

```Gherkin
Сценарий: Пример с группами
    * Группа шагов 1
        Дано шаг внутри группы
        И ещё шаг внутри группы
        * Вложенная группа
            Дано шаг во вложенной группе
```

**Правила:**
- Группы обозначаются отступами (табуляция или пробелы, но не смешивать)
- Строка группы не выполняется — выполняются только вложенные шаги
- Рекомендуется начинать имя группы с `*`
- Количество уровней вложенности не ограничено

Важно соблюдать отступы шагов внутри группы шагов. Правильно:
```Gherkin
* Закрытие всех окон перед началом
	И я закрываю все окна клиентского приложения

```
Неправильно, когда шаг внутри группы и сама группа находятся на одном уровне:
```Gherkin
* Закрытие всех окон перед началом
И я закрываю все окна клиентского приложения

```

## Подсценарии

### Объявление экспортного сценария

Чтобы сценарий можно было вызывать из других файлов, необходим тег `@ExportScenarios`:

```Gherkin
@ExportScenarios
Функционал: Экспортные сценарии

Сценарий: Я вызываю подсценарий с параметром "Парам1"
    Дано я вывожу значение "Парам1"
    И я вывожу "Текст[Парам1]вставлен"
```

**Объявление параметров в подсценарии:**
- Параметр в имени сценария: `"Парам1"`
- В шагах параметр используется как `"Парам1"` или `[Парам1]` (внутри параметров шагов подсценария)

### Вызов подсценария

```Gherkin
Сценарий: Основной сценарий
    Дано я вызываю подсценарий без параметров
    И я вызываю подсценарий с параметром "ЗначениеПараметра"
```

### Передача таблиц в подсценарии

Таблицы Gherkin можно передавать в подсценарии:

```Gherkin
Сценарий: Вызов с заменой таблиц
    И я вызываю подсценарий и передаю в него таблицу
    | Колонка1 | Колонка2 |
    | Знач1    | Знач2    |
```

**Особенности:**
- Таблицы пишутся сразу после строки вызова подсценария
- Можно передавать только часть таблиц, пропуская пустыми: `||`

### Передача многострочных строк в подсценарии

```Gherkin
Сценарий: Вызов с многострочным параметром
    И я вызываю подсценарий с параметрами
    """
    Многострочная
    строка
    """
```

## Управляющие конструкции

### Условия

```Gherkin
Сценарий: Пример условия
    Если текущее окно имеет заголовок "Заголовок" Тогда
        Дано шаг выполняется при истинности условия
    ИначеЕсли условие2 Тогда
        Дано альтернативный шаг
    Иначе
        Дано шаг по умолчанию
```

**Особенности:**
- Шаг-условие должен начинаться с `Если` и заканчиваться `Тогда`
- Любой шаг можно использовать как условие, добавив `Если...Тогда`
- Если шаг в условии вызывает исключение — результат `Ложь`
- Рекомендуется избегать условий там, где можно обойтись без них
- Если пользователь специально не просит, то не нужно добавлять оператор 'КонецЕсли'

### Циклы

```Gherkin
Сценарий: Пример цикла
    Дано для каждой строки таблицы "ИмяТаблицы" я выполняю
        И шаг внутри цикла
        Если условие Тогда
            Тогда я продолжаю цикл      // аналог Continue
        И я прерываю цикл              // аналог Break
```

**Шаги управления циклом:**
- `Тогда я продолжаю цикл` — аналог `Continue`
- `Тогда я прерываю цикл` — аналог `Break`

**Рекомендация:** Избегайте циклов там, где можно обойтись без них.
**Уточнение:** Шаги в теле цикла должны быть отделены дополнительным отступом от шага, который задаёт цикл.

### Попытка/Исключение

```Gherkin
Сценарий: Обработка исключений
    Попытка
        Дано шаг который может вызвать ошибку
        И ещё один рискованный шаг
    Исключение
        И я регистрирую ошибку "Описание" по данным исключения
```

**Особенности:**
- Состоит из ключевых слов `Попытка` и `Исключение`
- Шаги в каждой секции выделяются отступами
- При ошибке в секции `Попытка` выполнение переходит в `Исключение`
- Шаг `И я регистрирую ошибку...` доступен только в секции `Исключение`

## Специальные символы и экранирование

### Таблица экранирования

| Символ | Экранирование | Описание |
|--------|---------------|----------|
| Кавычка внутри кавычек | `\"` | `Часть1 \" Часть2` |
| Кавычка внутри апострофов | не требуется | `'Часть1 " Часть2'` |
| Апостроф | `\'` | `Часть1 \' Часть2` |
| Вертикальная черта в таблице | `\|` | `Текст \| Текст` |
| Обратная косая черта | `\\` | `\\\\ИмяСервера` |
| Фигурные скобки | `\{` `\}` | `\{СтрокаВФигурныхСкобках\}` |
| Перевод строки | `\n` | `Строка1\nСтрока2` |

### Правила экранирования

1. **Внутри кавычек `"`** — необходимо экранировать и кавычки, и апострофы
2. **Внутри апострофов `'`** — кавычки экранировать не нужно (приоритет апострофа)
3. **В таблицах Gherkin** — вертикальную черту `|` нужно экранировать
4. **Обратная косая черта** — для передачи одной `\` пишется `\\`
5. **Фигурные скобки** — экранируются если включено вычисление выражений

Вопрос: Как устроена секция переменных в feature файле?
Ответ: Вот описание как работать с секцией "Переменные".

### Секция "Переменные". Если есть возможность не использовать секцию переменные, то лучше её не использовать.

**Рекомендация:**
Использовать секцию "Переменные" только если об этом явно просит пользователь.

Секция `Переменные:` позволяет объявить переменные, доступные всем сценариям файла:

```Gherkin
Переменные:
    Перем1 = "СтрокаВКавычках"
    Перем2 = 'СтрокаВАпострофах'
    Перем3 = {Строка(1+1)}
    Перем4 = {!Строка(1+1)}
    ПростаяПеременная = "Значение"
    ВычисляемаяПеременная = {!Строка(1+1)}
    ЧерезДругуюПеременную = "Часть{ПростаяПеременная}Часть"
```
Переменные из секции переменных вычисляются перед выполнением каждого сценария.

**Типы значений переменных:**

| Тип | Описание | Пример |
|-----|----------|--------|
| Строка в кавычках | Обычная строка | `"Значение"` |
| Строка в апострофах | Строка с повышенным приоритетом | `'Значение'` |
| Многострочная строка | Строка с переносами | См. ниже |
| Выражение на клиенте | Вычисляется на клиенте | `{Выражение}` |
| Выражение на сервере | Вычисляется на сервере | `{!Выражение}` |

**Многострочные строки:**

```Gherkin
Переменные:
    Многострочная =
        """
        Первая строка
        Вторая строка
        Третья строка
        """
```          

### Вычисление выражений в секции переменных

Выражения в фигурных скобках вычисляются на стороне менеджера тестирования:

```Gherkin
{Выражение}      // вычисление на клиенте
{!Выражение}     // вычисление на сервере (! — признак сервера)
```

**Настройка:** Для вычисления выражений должна быть включена опция "Вычислять выражения в фигурных скобках".


### Импорт переменных из секции переменных

Есть возможность выполнить импорт переменных из секции переменных из другого фича файла
```Gherkin
Переменные:
    <Импорт> $КаталогИнструментов$\features\Файл1.feature
```


**Использование переменных из секции переменных в шагах:**

в параметр шага передали переменную
```Gherkin
Дано я вывожу значение переменной "Перем1"
```

переменная внутри строки
```Gherkin
И я вывожу "Часть{Перем2}Часть" 
```

### Табличные переменные в секции переменных

В секции переменных можно объявлять таблицы:

```Gherkin
Переменные:
    | Имя         | Возраст | Город    |
    | Иванов      | 30      | Москва   |
    | Петров      | 25      | СПб      |
    
    * ИменованнаяТаблица
    | Код | Наименование |
    | 001 | Товар1       |
    | 002 | Товар2       |
```

**Правила:**
- Таблица без имени может быть только одна
- Именованных таблиц может быть много
- Имя таблицы предваряется символом `*`
- В таблице должно быть 2 или более строк
- Первая колонка — уникальное имя строки

**Обращение к переменным таблицы:**

```Gherkin
// Для таблицы без имени:
Дано я вывожу "ИмяПеременной.Колонка"

// Для именованной таблицы:
Дано я вывожу "ИмяТаблицы.ИмяСтроки.Колонка"
```

## Введение

**Переменные** в Turbo Gherkin предназначены для хранения промежуточных значений, которые могут быть использованы:
- В пределах одного сценария
- В других сценариях того же файла
- В сценариях других файлов (для глобальных переменных)

---

## Типы переменных

Переменные в Turbo Gherkin бывают трёх типов:

| Тип | Описание | Рекомендация |
|-----|----------|--------------|
| **Строка** | Текстовые данные | **Основной и рекомендуемый тип** |
| **Число** | Числовые данные | Использовать при необходимости |
| **Дата** | Дата и время | Использовать при необходимости |

**Рекомендация:** Основным и рекомендуемым к использованию типом является **строковый тип**. Остальные типы следует использовать только при необходимости.

### Примеры получения переменных разных типов

```Gherkin
// Строка в переменную
Дано я запоминаю значение строки "Привет" в переменную "Текст"

// Число в переменную (через выражение)
И я вычисляю выражение на встроенном языке "10 + 20" и сохраняю в переменную "Сумма"

// Дата в переменную (через выражение)
И я вычисляю выражение на встроенном языке "ТекущаяДата()" и сохраняю в переменную "Дата"

// Дата как строка
И я вычисляю выражение на встроенном языке "Формат(ТекущаяДата(), 'ДФ=дд.ММ.гггг')" и сохраняю в переменную "ДатаСтрока"
```

## Локальные и глобальные переменные

### Локальные переменные

**Локальные переменные** существуют только в пределах сценария, в котором они были созданы.

**Особенности:**
- Создаются без специальных символов
- При использовании в шагах имя локальной переменной обрамляется **одним символом доллара**: `$ИмяПеременной$`
- После завершения выполнения сценария все локальные переменные автоматически удаляются
- Доступны только в текущем сценарии

```Gherkin
Сценарий: Пример с локальной переменной
    Дано я запоминаю значение строки "ООО Ромашка" в переменную "ИмяКонтрагента"
    И из выпадающего списка с именем 'Контрагент' я выбираю по строке "$ИмяКонтрагента$"
```

### Глобальные переменные

**Рекомендация:**
Использовать глобальные переменные только при необходимости

**Глобальные переменные** существуют, пока открыта форма Vanessa Automation.

**Особенности:**
- При создании имя переменной обрамляется **двумя символами доллара**: `$$ИмяПеременной$$`
- При использовании в шагах имя глобальной переменной также обрамляется **двумя символами доллара**: `$$ИмяПеременной$$`
- Доступны во всех сценариях, пока открыта форма
- Рекомендуется удалять перед созданием

```Gherkin
Сценарий: Пример с глобальной переменной
    Дано я удаляю глобальную переменную "ИмяКонтрагентаГлобально"
    И я запоминаю значение строки "ООО Ромашка" в переменную "$$ИмяКонтрагентаГлобально$$"
    
Сценарий: Использование глобальной переменной в другом сценарии
    Дано я вывожу значение переменной "$$ИмяКонтрагентаГлобально$$"
```

Вопрос: Как проверить равенство табличного документа эталону
Ответ: Для сравнения табличного документа с эталоном используются специальные шаги. Основной шаг имеет два параметра:
- **ИмяРеквизита** — имя реквизита табличного документа, как оно задано в конфигураторе
- **ИмяМакета** — имя файла эталонного макета (указывается без расширения)

**Требования к файлу эталона:**
1. Файл эталона должен быть в формате **mxl**
2. Файл эталона должен находиться в каталоге проекта либо в каталоге рядом с текущим фича-файлом

**Исключение ячеек из сравнения:**
- Для полного исключения ячейки укажите символ `*` в значении ячейки файла эталона
- Для частичного исключения значения замените исключаемую часть символом звёздочки и используйте шаг сравнения с эталоном по шаблону (использует регулярные выражения)

**Сравнение с таблицей Gherkin:**
Существуют шаги для сравнения табличного документа с таблицей Gherkin (обычное и по шаблону). Эти шаги следует использовать только для небольших макетов, так как размещение большого макета в сценарии ухудшает читаемость.

**Примеры на Gherkin:**
```Gherkin
* Основной вариант
  Дано Табличный документ "ИмяРеквизита" равен макету "ИмяМакета"
  Дано Табличный документ "ИмяРеквизита" равен макету "ИмяМакета" по шаблону

* Более редкий вариант (сравнение с таблицей Gherkin)
  Тогда табличный документ "РеквизитТабличныйДокумент" равен:
    | 'Значение1' | 'Значение2' |
    | 'Значение3' | '*'         |
  
  Тогда табличный документ "РеквизитТабличныйДокумент" равен по шаблону:
    | 'Значение1' | 'Значение2' |
    | 'Значение3' | 'Знач*ние4' |
```

Вопрос: Почему значение в отчете отличается от введенного в документе?
Ответ: В отчетах числовые значения часто отображаются с фиксированным форматом (например, 10,000 вместо 10). При проверке ячеек табличного документа используйте точное значение с форматом, который отображается в отчете.

**Пример:**
```Gherkin
// Ввели количество 10 в документе
И в таблице "Товары" в поле с именем 'ТоварыКоличествоУпаковок' я ввожу текст "10"

// В отчете значение будет с форматом 10,000
И в табличном документе "ОтчетТабличныйДокумент" ячейка с адресом "R11C4" равна "10,000"
```

**Важно:** При проверке ячеек отчета для числовых значений нельзя использовать проверку через переменные с шаблоном. Нужно использовать прямой шаг проверки ячейки:
```Gherkin
И в табличном документе "ОтчетТабличныйДокумент" ячейка с адресом "R11C4" равна "10,000"
```

Вопрос: Как в сценарии выполнить код на языке 1С
Ответ: Помещение кода на встроенном языке 1С внутрь сценария приводит к ухудшению читаемости, поэтому эту возможность следует использовать только в исключительных случаях.
Предпочтительнее создавать новый шаг и помещать код на 1С внутрь этого шага.

**Варианты выполнения кода на стороне менеджера тестирования:**
1. Выполнить клиентский код
2. Выполнить серверный код с возможностью обращения к объекту Vanessa Automation
3. Выполнить серверный код без возможности обращения к объекту Vanessa Automation (предпочтительный вариант, если нужно просто вызвать серверный код тестируемой конфигурации при условии, что менеджер тестирования запущен в той же базе, что и клиент тестирования)

**Варианты выполнения кода на стороне клиента тестирования:**
Для этого в базе клиента тестирования должно быть установлено расширение **VA Extension**. Доступны два варианта:
1. Выполнить клиентский код с помощью расширения
2. Выполнить серверный код с помощью расширения

**Примеры на Gherkin:**
```Gherkin
* Код выполняется на клиенте (менеджер тестирования)
  И я выполняю код встроенного языка
  """bsl
    Сообщить("Hello world.")
  """

* Код выполняется на сервере с контекстом Vanessa Automation
  И я выполняю код встроенного языка на сервере
  """bsl
    Сообщить("Hello world.")
  """
```

Вопрос: Как правильно проверить видимость/доступность кнопки, элемента или пункта меню
Ответ: Существуют стандартные шаги проверки доступности и видимости элементов формы. Однако они проверяют не реальную видимость/доступность, а специальные флаги элементов формы.

**Важная особенность:**
У элемента формы может стоять флаг видимости, но при этом он может быть реально не видим пользователю (например, если элемент скрыт через условное оформление или с помощью функциональной опции).
Поэтому стандартные шаги следует использовать только если не используются подобные механизмы скрытия.

**Альтернативные подходы:**

1. **Специальный шаг для проверки негативного поведения** — сначала проверьте, что кнопка существует, затем вызовите проверяемый шаг в специальном шаге для проверки негативного сценария

2. **Конструкция Попытка/Исключение** — проверьте, что кнопка существует, затем выполните действие с элементом в блоке Попытка, в блоке Исключение обработайте ожидаемое исключение

3. **Проверка через условие** — попытайтесь выполнить действие с полем (например, очистить его). Если действие успешно выполнено — элемент доступен, если нет — вызовите исключение

**Примеры на Gherkin:**
```Gherkin
* Проверка доступности (стандартный шаг)
  И элемент формы с именем "ИмяЭлемента" доступен

* Проверка видимости (стандартные шаги)
  И я жду появления элемента "Заголовок элемента" в течение 20 секунд
  И элемент формы "Заголовок поля" существует и невидим на форме

* Проверка доступности через специальный шаг (негативное поведение)
  И кнопка с именем "ИмяКнопки" существует
  И Проверяю шаги на Исключение:
      |'И я нажимаю на кнопку с именем "ИмяКнопки"'|

* Проверка доступности через конструкцию Попытка/Исключение
  И кнопка с именем "ИмяКнопки" существует
  Попытка
    И я нажимаю на кнопку с именем "ИмяКнопки"
    И я вызываю исключение "Ожидалось, что кнопка недоступна пользователю."
  Исключение
    И Пауза 0.1

* Проверка доступности поля через условие
  И поле с именем "ИмяПоля" существует
  Если я нажимаю кнопку очистить у поля с именем "ИмяПоля" тогда
    И я вызываю исключение "Поле оказалось доступно"
```

Вопрос: Особенности шагов при работе с переключателями, гиперссылками и полями
Ответ: Для переключателей, гиперссылок и полей существует два способа задания значения:

**Работа с переключателями:**

1. **По представлению** — текстовое значение
2. **По номеру** — порядковый номер значения в списке (параметр типа Число)

**Примеры на Gherkin:**
```Gherkin

  И у поля с именем "ИмяЭлемента" я нажимаю гиперссылку 'ИмяГиперссылки'
  И у поля с именем "ИмяЭлемента" я нажимаю гиперссылку 1

* Поля (по представлению и по номеру)
  И из выпадающего списка с именем "ИмяЭлемента" я выбираю точное значение "Значение"
  И из выпадающего списка с именем "ИмяЭлемента" я выбираю точное значение 1
```

Вопрос: Когда надо использовать паузу
Ответ: При тестировании приложений на платформе 1С:Предприятие нужно учитывать асинхронность: действие в форме может ещё не успеть выполниться, а контекст управления уже возвращается в менеджер тестирования.
Это происходит при использовании обработчиков ожидания, отложенном проведении документов, отложенной инициализации форм и т.д.

**Рекомендации:**
- Если есть возможность использовать шаги ожидания конкретного события (закрытия окна, заполнения табличного документа данными и т.д.) — используйте их
- Шаг **Пауза** следует использовать только когда нет возможности ожидать определённого события или состояния
- Старайтесь избегать больших пауз, так как это приводит к существенному замедлению выполнения сценария

**Примеры на Gherkin:**
```Gherkin
* Шаги ожидания конкретного события (предпочтительно)
  И я жду закрытия окна "Нужное окно" в течение 20 секунд
  И я жду, что табличный документ "ИмяЭлемента" заполнится данными в течение 10 секунд
  И выражение внутреннего языка 'Выражение' стало Истинно в течение 10 секунд
  И я жду, что в таблице "ИмяТаблицы" количество строк будет "больше" 0 в течение 20 секунд

* Шаг паузы (использовать только когда нет возможности ожидать события)
  И пауза 5
  И пауза 0.1
```

Вопрос: Выбор файла из диалога выбора файлов

и не правильно делать так:

Вопрос: Когда можно использовать шаг «Пауза»?
Ответ: Шаг «Пауза» нужно использовать только в тех случаях, где без него никак нельзя. Во всех случаях, в которых есть какое-нибудь событие (появление окна, изменение доступности или значения поля и т.д.), которое означает завершение ожидания, необходимо использовать шаг ожидания данного события.

**Примечание:** Если нужна большая пауза, так как появляется окно «Пожалуйста, подождите», то можно использовать следующие шаги:
```Gherkin
И пауза 3
И я жду закрытия окна "" в течение 20 секунд
```
**Пояснение:** В этом примере шаг паузы используется не как замена ожиданию, а как необходимая часть комбинации:
1. `И пауза 3` — небольшая пауза, чтобы окно успело появиться (окно может появиться с задержкой)
2. `И я жду закрытия окна "" в течение 20 секунд` — ожидание закрытия окна

Такая комбинация шагов не отменяет общую рекомендацию избегать пауз, а является допустимым исключением для специфического сценария, когда окно появляется с задержкой.

Если окно не открылось, то шаг ожидания все равно отработает корректно, т.к. будет считать, что окно уже закрылось.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как рекомендуется проверять значения в регистрах для теста?
Ответ: Если для теста необходимо проверить значения в регистрах, то рекомендуется проверять их с помощью отчетов. Так как регистры больше предназначены для внутреннего администрирования, их структура изменяется чаще, чем стандартные отчеты. Поэтому при использовании проверок по регистрам придется больше времени тратить на актуализацию автотеста при изменениях в поведении конфигурации.

Вопрос: Как правильно нажимать на гиперссылки в тестах?
Ответ: Нужно использовать шаги
Для одиночных гиперссылок
```Gherkin
И я нажимаю на гиперссылку "Гиперссылка"
И я нажимаю на гиперссылку с именем 'Гиперссылка'
```

Для гиперссылок встроенных в поле

```Gherkin
И у поля с именем 'ИмяПоля' я нажимаю гиперссылку 'ИмяГиперссылки'
```

Полный список шагов для работы с гиперссылками можно получить поиском по типу шага "UI.Формы.Поля.Гиперссылки"

Вопрос: Как планировать создание группы сценариев с подсценариями?
Ответ: При планировании создания группы сценариев необходимо заранее планировать подсценарии, вынося в них общие для разных сценариев части. Это избавит от необходимости рефакторинга тестов в будущем, и тесты будет легче поддерживать – при изменении поведения конфигурации в одном месте не нужно будет править несколько сценариев, только один подсценарий.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как структурировать разные части сценария?
Ответ: Для структуризации разных частей сценария рекомендуется группировать шаги в группы. Наименования групп указываются с звездочкой в начале (например, `*НазваниеГруппы`). Группы позволяют определить, какой именно смысл лежит в действиях тех или иных шагов, так как смысл накликать невозможно.

Вопрос: Как работать с переключателями, представления значения которых постоянно изменяются? Например, в представлении содержится текущая дата.
Ответ: При использовании переключателей, значения которых часто изменяются (например, значение в скобках будет часто изменяться), чтобы не тратить много времени на актуализацию таких сценариев, рекомендуется использовать переход к значению переключателя по шаблону.

**Пример:**
```Gherkin
И я меняю значение переключателя "Таблица" на 'Прочих расходов (*)' по шаблону
```

Вопрос: Что делать, если у гиперссылки может постоянно изменяться заголовок? Например, в представлении содержится текущая дата.
Ответ: Если у гиперссылки может постоянно изменяться заголовок (например, в нем присутствует текущая дата), то можно обращаться к гиперссылке по её порядковому номеру у соответствующего поля.

**Пример:**
```Gherkin
И у поля с именем "СчетФактураПредставление" я нажимаю гиперссылку 0
```

Вопрос: Что нужно учитывать при использовании шага `И я нажимаю кнопку выбора у поля с именем "ИмяПоля"`?
Ответ: Нужно учитывать, что перед выбором шаг выполняет активизацию поля. То есть внутри шага зашиты два действия – активизация поля и нажатие кнопки выбора. При этом иногда разработчики добавляют действие и на активизацию поля выбора, и тогда уже на этапе активизации проходит открытие формы выбора, и после выполнения шага открывается две формы выбора.

**Решение:** В таких случаях необходимо заменить шаг выбора на шаг активации:
```Gherkin
И я активизирую поле с именем "ИмяПоля"
```

Вопрос: Как правильно выбрать элемент в поле?
Ответ: При необходимости выбора элемента в поле лучше использовать выбор по строке, а не форму подбора или форму выбора. С использованием форм тест становится длиннее и начинает зависеть от работы формы.

**Выбор по строке выполняется с помощью шага:**
```Gherkin
И из выпадающего списка с именем "ИмяПоля" я выбираю по строке "ЧастьСтроки"
```

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Что нужно делать после проведения документа из формы списка?
Ответ: Если выполняется проведение документа из формы списка, необходимо выполнить проверку, что не появились окна с предупреждениями.

**Пример:** Если используется шаг:
```Gherkin
И в таблице "СписокДокументыПродажи" я нажимаю на кнопку с именем 'СписокПровести'
```
то после него нужно добавить шаг:
```Gherkin
Тогда не появилось окно предупреждения системы
```

Вопрос: Как «долистать» значение у поля формы до нужного, если заранее неизвестно сколько раз нужно выполнить действие?
Ответ: Иногда у полей есть кнопки, которые меняют их значение (увеличивают или уменьшают). И бывает заранее неизвестно, сколько раз нужно выполнить это действие, а известно лишь конечное значение.

**Решение:**
```Gherkin
Цикл И я делаю 100 раз
    И Действие переключение значения поля
    Если поле с именем "ИмяПоля" имеет значение "НужноеЗначение" тогда
        Тогда я прерываю цикл
```

Вопрос: Что делать, когда в поле был введён текст, ошибки не возникло, затем проверка значения поля показывает, что текст вернулся к прежнему значению?
Ответ: Эта проблема связана с тем, что после ввода текста в форме произошло окончание контекстного серверного вызова. Это привело к тому, что состояние формы на клиенте было возвращено к состоянию формы сервера.

**Решения:**

**На уровне конфигурации:** Полю формы указать значение «ОбновлениеТекстаРедактирования» в «При изменении значения».

**На уровне сценария:**

- Для одного поля (рекомендуется):
```Gherkin
И я отключаю обновление текста редактирования поля с именем "ИмяПоля" в текущем окне (расширение)
```

- Для всех полей формы:
```Gherkin
И я отключаю обновление текста редактирования полей в текущем окне (расширение)
```

- Сделать, чтобы Vanessa автоматически изменяла это свойство для всех полей всех форм:
```Gherkin
И я устанавливаю флаг настройки Vanessa Automation "ОтключатьДляПолейВозможностьЗатиранияТекстаПриСерверномВызове"
```

Вопрос: Как правильно использовать выбор из панели разделов и панели функций?
Ответ: При работе с панелью разделов правильно использовать один шаг, который выполняет выбор из панели разделов и панели функций вместо двух. Это связано с тем, что один шаг будет внутри выполнять несколько попыток добиться нужного результата, а два шага – нет.

**Правильно:**
```Gherkin
И В командном интерфейсе я выбираю "Бюджетирование и планирование" "Настройки и справочники"
```

**Неправильно:**
```Gherkin
И В панели разделов я выбираю "Бюджетирование и планирование"
И В панели функций я выбираю "Настройки и справочники"
```

Вопрос: Когда можно использовать шаг выполнения кода встроенного языка?
Ответ: Шаги вида:
```Gherkin
И я выполняю код встроенного языка
    | 'МояПроцедура()' |
```
нужно использовать только в том случае, если выполняется не сложный код. Если нужно выполнить какой-либо сложный алгоритм, передать значение с сервера на клиент и т.п. – нужно создать новый шаг для Vanessa Automation и использовать его.

Вопрос: Что нужно учитывать при использовании шага, выбирающего контекстное меню таблицы?
Ответ: При использовании шага, выбирающего контекстное меню таблицы, нужно, чтобы в таблице была активная строка. Если активной строки нет, то пункты меню могут быть недоступны.

Вопрос: Можно ли использовать шаг `И я закрываю текущее окно`?
Ответ: Лучше не использовать шаг:
```Gherkin

```
т.к. это может приводить к ложным срабатываниям. Лучше использовать шаг:
```Gherkin
И я закрываю окно "Заголовок окна"
```

Вопрос: Что нужно учитывать при открытии нескольких клиентов тестирования в сценарии?
Ответ: Если в сценарии открывается несколько клиентов тестирования, то нужно рассчитывать объём оперативной памяти, который они будут занимать. Например, один сеанс ERP может занимать несколько гигабайт памяти.

**Рекомендация:** Если есть возможность, то лучше закрывать неиспользуемые сеансы клиентов тестирования во время выполнения сценариев. Если они нужны несколько раз во время выполнения сценария, то их лучше закрывать и подключать заново, когда они понадобятся.

Вопрос: Нужно ли фиксировать значение в поле перед использованием паузы?
Ответ: Нужно зафиксировать значение в поле, прежде чем использовать паузу. Если в сценарии происходит изменение значения в поле на форме, то важно завершить редактирование поля перед тем, как вызывать паузу. Иначе есть риск, что пока редактирование поля не завершено, произойдёт контекстный серверный вызов, и значение поля будет возвращено к предыдущему.

**Правильно:**
```Gherkin
И из выпадающего списка с именем "ХозяйственнаяОперация" я выбираю точное значение 'Закупка по регл. учету'
И я перехожу к следующему реквизиту
И пауза 5
```

**Неправильно:**
```Gherkin
И из выпадающего списка с именем "ХозяйственнаяОперация" я выбираю точное значение 'Закупка по регл. учету'
И пауза 5
```

Вопрос: Что такое таблица клиентов тестирования и как она устроена?
Ответ: Таблица клиентов тестирования — это таблица на закладке "Настройки клиентов тестирования", в которой задаются настройки для подключения клиентов тестирования. Каждая строка таблицы — это набор настроек для подключения конкретного клиента тестирования. Таблицу можно заполнить вручную, во время выполнения сценария или передать в JSON-файле при запуске тестов из командной строки. Первой строкой в таблице всегда будет специальная служебная строка, которая соответствует той базе, в которой запущена Vanessa Automation. У этой строки есть специальная пометка "Признак служебной строки". У других строк такой пометки не будет.

Вопрос: Какие колонки есть в таблице клиентов тестирования и что они означают?
Ответ:
- Имя — колонка, в которой задаётся имя подключения. Имя должно быть уникально в пределах всей таблицы. Имя подключения важно, так как позволяет подключать и отключать клиентов тестирования по их имени.
- Синоним — колонка, в которой задаётся синоним подключения. Эта колонка в основном используется, когда Vanessa Automation работает в связке с СППР. При работе из СППР имя подключения генерируется автоматически, поэтому его не получится использовать в сценарии, так как оно заранее неизвестно. В этом случае можно использовать колонку "Синоним" как альтернативное имя подключения.
- Тип клиента — колонка, в которой выбирается тип клиента тестирования.
- Путь к инфобазе — колонка, в которой указывается путь к базе, которая будет выступать в роли клиента тестирования.
- Дополнительные параметры — колонка, в которой указываются дополнительные параметры подключения (имя пользователя, пароль и другие ключи запуска).
- Имя компьютера — колонка, в которой задаётся имя компьютера, на котором находится уже запущенный клиент тестирования, либо где будет запускаться новый клиент тестирования.
- Порт — колонка, в которой указывается порт для запуска или подключения клиента тестирования.
- PID клиента — идентификатор процесса (заполняется при использовании компоненты VanessaExt).

Вопрос: Как происходит поиск строки в таблице клиентов тестирования по имени подключения?
Ответ: Все шаги, которые ищут строку в таблице клиентов тестирования по имени подключения, сначала выполняют поиск по колонке "Имя", а потом по колонке "Синоним".

Вопрос: Какие типы клиентов тестирования доступны в Vanessa Automation?
Ответ: Доступны следующие типы клиентов:
- Тонкий — сеанс 1С на управляемых формах
- Толстый — также сеанс 1С на управляемых формах
- Web — сеанс 1С в браузере
- Обычное приложение — сеанс 1С на обычных формах
- Мобильный клиент
- Мобильное приложение

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как заполняется колонка "Путь к инфобазе" для разных типов клиентов?
Ответ:
- Для тонкого и толстого клиента: либо путь к файловой базе (например: File="c:\Bases\MyBase";), либо строка подключения к серверной базе (например: Srvr="MyServer";Ref="MyBase";).
- Для Web клиента: ссылка, которая используется в браузере (например: http://localhost/MyBase).

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Что указывается в колонке "Дополнительные параметры"?
Ответ: В этой колонке указываются дополнительные параметры подключения. Обычно это имя пользователя, пароль и другие ключи запуска (например: /N"Администратор" /P"Пароль" /ДопКлючи).

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как заполняется колонка "Имя компьютера" для разных типов клиентов?
Ответ:
- Для тонкого и толстого клиента: имя компьютера, на котором находится уже запущенный клиент тестирования, либо где будет запускаться новый клиент тестирования. Если клиент тестирования запускается на том же ПК, что и менеджер тестирования, то в настройке надо указать значение localhost.
- Для Web клиента (файловая база): значение localhost.
- Для Web клиента (серверная база): сетевое имя кластера сервера 1С или его IP-адрес. При этом важно, чтобы менеджер тестирования имел доступ по локальной сети к менеджеру кластера. Иначе подключить клиент тестирования не получится.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как заполняется колонка "Порт" для разных типов клиентов?
Ответ:
- Для тонкого и толстого клиента: если нужно запустить новый клиент тестирования, то можно указать значение свободного порта на данном ПК или просто указать значение 1. Если нужно подключиться к уже запущенному клиенту тестирования, то надо указать реальное значение порта, на котором ранее был запущен клиент тестирования.
- Для Web клиента (файловая база): значение порта локального web-сервера. Обычно это 1538.
- Для Web клиента (серверная база): порт кластера серверов 1С. Обычно это 1541.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как определяется уникальность клиента тестирования для разных типов клиентов?
Ответ:
- Для тонкого и толстого клиента: уникальность определяется именно по порту. Другие поля не учитываются.
- Для Web клиента: уникальность определяется по имени подключения. Другие поля не учитываются.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как правильно работать с портами при запуске клиентов тестирования?
Ответ: При запуске нового клиента тестирования Vanessa Automation будет проверять список доступных портов в системе и выберет тот, который сейчас свободен. Если указать порт неправильно, то подключиться к ранее запущенному клиенту не получится. Есть возможность указать диапазон портов, чтобы порт выбирался из него (настройка "ДиапазонПортовTestclient").

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как корректно завершить сеанс Web клиента?
Ответ: При завершении сеанса Web клиента нельзя закрывать вкладку браузера или сам браузер. В этом случае сеанс Web клиента зависнет на сервере. Чтобы корректно завершить сеанс Web клиента, надо нажать в меню: Файл, затем — Выход. Тогда сеанс Web клиента завершится корректно.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: При записи действий при работе с модальным диалогом появился шаг: И я нажимаю на кнопку с именем 'Button0'. Корректно ли использовать такой шаг, ведь по имени кнопки не понятно смысл действия.
Ответ: Да, при работе с модальными диалогами имена кнопок очень часто не несут смысла: Button0, Button1, Button2 и так далее.
Поэтому для модальных диалогов лучше использовать шаги
```Gherkin
И я нажимаю на кнопку 'ОК'
```
или
```Gherkin
И я нажимаю на кнопку 'Да'
```
в зависимости от того какая кнопка была нажата.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Правильно ли в таблице в поле ввести текст, а затем вызвать завершение редактирования строки.
Ответ:
Видимо речь идёт о таком случае, когда идёт попытка использовать такой набор шагов:
```Gherkin
    И в таблице "Товары" я нажимаю на кнопку с именем "ТоварыДобавить"
    И в таблице "Товары" в поле "Номенклатура" я ввожу текст "Материал без характеристики 1"
    И В таблице "Товары" я завершаю редактирование строки
```

**Важно:** Данный вопрос актуален только когда заполняется последнее редактируемое поле в строке таблицы.

**Если после ввода значения в поле курсор автоматически переходит в следующее поле строки**, то результат редактирования сохраняется автоматически при переходе. В этом случае шаг завершения редактирования строки не требуется.

**Если же заполняется последнее поле в строке** и после ввода значения курсор не переходит в другое поле, то для сохранения значения необходимо вызвать шаг завершения редактирования строки:
```Gherkin
И в таблице "Товары" в поле с именем 'ТоварыЦена' я ввожу текст "100"
И в таблице "Товары" я завершаю редактирование строки
```

**Когда нужно вызывать завершение редактирования строки:**

| Ситуация | Нужно ли завершение редактирования |
|----------|-----------------------------------|
| Заполнено не последнее поле в строке таблицы | ❌ Нет, курсор сам перейдёт в следующее поле |
| Заполнено последнее поле в строке таблицы | ✅ Да, иначе значение не сохранится |

**Примечание:** Шаг "завершение редактирования строки" существует только для таблиц. Для полей шапки формы достаточно перехода к следующему реквизиту.

Так можно делать для реквизитов простых типов; строка, число, дата.

Если же реквизит имеет ссылочный тип, например это Справочник, Документ, Перечисление, то в таком случае ввод значения в поле не произойдёт и для таких типов надо использовать шаг:
- для справочников, документов и других ссылочных типов нужно использовать ввод значения по строке. Пример:
```Gherkin
И в таблице "Товары" из выпадающего списка с именем "Организация" я выбираю по строке "Торговая компания"
```
- для перечислений лучше использовать выбор из списка точного значения. Например:
```Gherkin
И в таблице "Товары" из выпадающего списка с именем 'ЮрФизЛицо' я выбираю точное значение "Компания"
```
Но выбор значения по строке для перечислений тоже будет работать.
Эту логику можно использовать и при работе с полями шапки формы.

Вопрос: Как правильно генерировать шаги, которые заполняют поля значениями?
Ответ: Правильно делать так.
Надо создать шаг, который заполяет поле значением, затем добавить шаг, который делает переход в следующее поле, чтобы зафиксировать введенное значение, затем надо добавить проверку значения в исходном поле.
Пример:
```Gherkin
	И из выпадающего списка с именем "Организация" я выбираю по строке "Торговая компания"
	И я перехожу к следующему реквизиту
	Тогда элемент формы с именем 'Организация' стал равен "Торговая компания"
```
Если происходил ввод в поле даты, то надо учесть, что у полей типа Дата поле может быть содержать разный формат даты:
- только основная дата: dd.mm.yyyy
- основная дата плюс время: dd.MM.yyyy ss.mm.hh
- только время: ss.mm.hh
Поэтому может быть ситуация, что в поле дата ввели дату в формате "30.10.2025", а при проверке значения поля появится ещё и время.
В этом случае можно сделать так:
```Gherkin
	И в поле с именем 'Дата' я ввожу текст "30.10.2025"
	И я перехожу к следующему реквизиту
	Тогда элемент формы с именем 'Дата' стал равен "05.05.2026*" по шаблону
```

Вопрос: Что делать если падает шаг проверки закрытия окна?
Ответ: Допустим, падает шаг:
```Gherkin
И я жду закрытия окна "Нужное окно" в течение 20 секунд
```
Чтобы понять почему окно не закрылось, стоит получить текст сообщений пользователю. Часто там описана проблема, из-за которой окно не закрылось.

Вопрос: Упал шаг сценария. Я исправил этот шаг - лучше запустить сценарий с начала или запустить сценарий с текущего шага?
Ответ: Лучше запустить сценарий с текущего шага, т.к. сценарии могут быть длинными. Если тест падает на строке 900 из 1000, то если есть возможность запустить сценарий не с самого начала, то лучше так и сделать.
При этом, если текст шага был исправлен напрямую в файле, нужно перезагрузить сценарий в Vanessa Automation.

Вопрос: Как правильно проверить условием значение флага?
Ответ: Нужно использовать специальные шаги, в зависимости от окружения.
В русском интерфейсе:
- проверка на Истина
```Gherkin
Если элемент формы с именем "ИспользоватьЗаказыПоставщикам" имеет значение "Да" Тогда
	....какие-то действия
```

- проверка на Ложь
```Gherkin
Если элемент формы с именем "ИспользоватьЗаказыПоставщикам" имеет значение "Нет" Тогда
	....какие-то действия
```

В английском интерфейсе:
- проверка на Истина
```Gherkin
Если элемент формы с именем "ИспользоватьЗаказыПоставщикам" имеет значение "Yes" Тогда
	....какие-то действия
```

- проверка на Ложь
```Gherkin
Если элемент формы с именем "ИспользоватьЗаказыПоставщикам" имеет значение "No" Тогда
	....какие-то действия
```

При этом регистр букв важен.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Если пользователь хочет убедиться, что флаг точно установлен или снят - как это реализовать?
Ответ: Для этого лучше использовать шаги установки или снятия флагов.
- для установки флага:
```Gherkin
И я устанавливаю флаг "ИмяФлага"
```
- для снятия флага
```Gherkin
И я снимаю флаг "ИмяФлага"
```
Если флаг уже установлен и вызвать шаг установки флага, то ничего не изменится.
Также если флаг снят и вызван шаг снятия флага, тоже ничего не изменится.
Поэтому достаточно вызвать шаги сятия или установки флага. Это будет достаточно, чтобы проверить и убедиться, что у флага нужное значение.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Как правильно использовать шаг: 'Если элемент "ИмяЭлементаФормы" присутствует на форме Тогда'?
Ответ: Нужно учесть, что если в параметр шага передать имя поля, которого вообще нет на форме, то шаг вернёт Ложь и не будет падать в исключение.
Поэтому перед сначала нужно убедиться, что поле существует на форме, при этом оно может видимо или невидимо, но главное, что оно точно есть.
Правильно делать так:
```Gherkin
И поле с именем 'ИмяЭлементаФормы' существует
Если элемент "ИмяЭлементаФормы" присутствует на форме Тогда
	... какие-то действия
```
Для кнопок похожая логика:
```Gherkin
И кнопка с именем 'ИмяКнопки' существует
Если элемент "ИмяКнопки" присутствует на форме Тогда
	... какие-то действия
```

Вопрос: Как можно закрыть окно клиента тестирования?
Ответ: Закрыть окно клиента тестирования можно несколькими способами:
- есть шаг, который закрывает окно по его заголовку:
```Gherkin
И Я закрываю окно "Заголовок окна"
```
- есть шаг, который закрывает текущее окно
```Gherkin
И я закрываю текущее окно
```
**Важно:** Так как в этом шаге явно не указано имя закрываемого окна, может возникнуть ситуация, что в какой-то момент поведение системы изменится и откроется другое окно. В этом случае шаг закроет не то окно, которое планировалось, а то, которое в данный момент является текущим. Поэтому рекомендуется использовать шаг с явным указанием заголовка окна.
- окно может закрыться, если нажать на кнопку "ЗаписатьИЗакрыть", "ПровестиИЗакрыть", "Закрыть" или их аналоги.
Но в таком случае важно убедиться, что такие кнопки есть на форме.
Если специальных таких кнопок на форме нет, то надо использовать предыдущие варианты.

Вопрос: Что может означать ошибка "Невидимый пользователю элемент управления не может выполнять интерактивные действия"?
Ответ: Эта ошибка точно означает, что пользователь в данный момент не видит элемент формы и не может с ним взаимодействовать.

**Важно:** В отличие от стандартных шагов проверки видимости (которые проверяют специальные флаги элементов формы), данная ошибка возникает при реальной невидимости элемента для пользователя.

Возможные причины:
- Элемент невидим пользователю
- Элемент недоступен для редактирования
- Элемент скрыт настройками функциональных опций
- Элемент скрыт условным оформлением
- Элемент находится в группе элементов, которая сейчас свернута (в этом случае можно попробовать развернуть группу, в которой находится элемент с помощью шага: И я разворачиваю группу с именем 'ИмяГруппы')

**Примечание:** Стандартные шаги проверки доступности и видимости элементов (например, "И элемент формы с именем 'ИмяЭлемента' доступен") проверяют специальные флаги элементов формы, а не реальную видимость. Поэтому их следует использовать только если не используются механизмы скрытия через условное оформление или функциональные опции.

------------------------------------------------------------------------------------------------------------------------------------------

Вопрос: Всегда ли в списке документов новые документы создаются с помощью кнопки "Создать"?
Ответ: Чаще всего да, но бывают исключения, когда в одном списке документов могут находится документы разных видов.
Тогда для создания каждого вида документа может быть своя кнопка, например: 'Создать___ЗакупкаУПоставщика___ПриобретениеТоваровУслуг'.

Вопрос: У текущего окна клиента тестирования в конце его заголовка находится символ '*', например 'Заявка на закупку (создание) *'. Как мне правильно выполнить закрытие такого окна?
Ответ: Символ '*' в конце заголовка окна означает, что у окна установлен признак модифицированности.
Поэтому при попытке закрыть его с помощью шагов
```Gherkin
И я закрываю текущее окно
```
или
```Gherkin
И я закрываю окно 'Заявка на закупку (создание) *'
```
пользователю будет показан модальный диалог с текстом: 'Данные были изменены. Сохранить изменения?'
Этот диалог является отдельным окном и его нужно отдельно обрабатывать, например так. Если нужно сохранить изменения в форме:
```Gherkin
Тогда появилось предупреждение, содержащее текст "Данные были изменены. Сохранить изменения?"
И я нажимаю на кнопку 'Да'
```
или так, если нужно отменить изменения в форме:
```Gherkin
Тогда появилось предупреждение, содержащее текст "Данные были изменены. Сохранить изменения?"
И я нажимаю на кнопку 'Нет'
```

Вопрос: В сеансе клиента тестирования появилось окно с заголовком '1С:Предприятие'. Как с ним правильно работать?
Ответ: Окно с заголовком '1С:Предприятие' - это модальный диалог.
Он блокирует окно, к которому относится модальный диалог.

```Gherkin
Тогда появилось предупреждение, содержащее текст "Текст"
```
или
```Gherkin
Если появилось предупреждение, содержащее текст "Текст" тогда
	// ... действия
```
Чтобы найти все шаги, которые могут работать с предупреждениями, нужно выполнить поиск шагов с помощью инструмента search_for_steps_by_keywords, search_name=предупр

Вопрос: В каких ситуациях могут появляться модальные диалоги с заголовком '1С:Предприятие'?

  - В данном документе нельзя выбирать данное значение в данное поле по 'такой-то причине'.
  - [другие варианты, они зависят от логики работы приложения и логики работы конкретной формы]

Вопрос: Как проверять значения полей таблиц, когда значение поля и текст редактирования различаются?
Ответ: В некоторых случаях значение поля формы и текст редактирования могут различаться. Это происходит, когда:
- Поле имеет тип Перечисление
- Отображаемое значение отличается от внутреннего значения
- Поле находится в таблице формы

Пример:
```Gherkin
# Поле имеет одно значение, но текст редактирования другой
И элемент формы с именем 'НалогообложениеНДС' стал равен "Продажа облагается НДС"
И у элемента формы с именем 'НалогообложениеНДС' текст редактирования стал равен "Закупка облагается НДС"
```

Рекомендация: Для проверки таких полей используйте шаг:
```Gherkin
Тогда у элемента формы с именем 'ИмяПоля' текст редактирования стал равен "Ожидаемое значение"
```

Вопрос: Как найти имя поля, если известно только значение текста редактирования?
Ответ: Используйте инструмент `get_form_analysis` с параметром `format=gherkin`. В результате будут показаны все элементы формы с их значениями и текстами редактирования.
```Gherkin
И у элемента формы с именем 'ИмяПоля' текст редактирования стал равен "Значение"
```

Вопрос: Как работает запуск сценария с указанного шага при работе инструмента run_scenario, mode=reloadAndRunFromLine' (перезагрузить фича файл и выполнить сценарий с указанной строки).
Ответ: В этом случае сначала фича файл будет перезагружен из файла. Затем будет попытка выполнить сценарий со строки, указанной в параметре lineNumber.
Пример. Допустим есть такой фича файл:
```Gherkin
#language: ru
Функционал: Тестовая фича
Сценарий: Тестовый сценарий

И В командном интерфейсе я выбираю "НСИ и администрирование" "Закупки"
И я жду открытия окна "Закупки" в течение 20 секунд
И я устанавливаю флаг с именем 'ИспользоватьЗаказыПоставщикам'
```

В этом примере первый шаг, который можно выполнить находится в пятой строке, это шаг 'И В командном интерфейсе я выбираю "НСИ и администрирование" "Закупки"'
Поэтому если указать lineNumber=5 сценарий будет выполнен начиная с этого шага.
Также корректно указать lineNumber=3 и lineNumber=4. Строки 3 и 4 не содержат шагов, которые можно выполнить, поэтому будет выполнен поиск первой строки, в которой есть шаг сценария.
Это будет строка 5. Поэтому сценарий также будет выполнен с пятой строки.

Вопрос: Я ввожу в поле значение и нужно сгенерировать шаг для проверки поля. Как это правильно сделать?
Ответ: Для разных типов полей нужно использовать разные подходы.

## 1. Для ссылочных полей с типом Справочник и Документ надо делать так:
```Gherkin
И из выпадающего списка с именем 'Контрагент' я выбираю по строке "ООО Ромашка"
И я перехожу к следующему реквизиту
Тогда элемент формы с именем 'Контрагент' стал равен "ООО Ромашка"
```
Т.е. что ввели в поле и проверяем, что значение стало таким же.

Важный момент: при вводе значения по строке нужно учитывать, что шаг, выбирающий значение мог вводить только часть строки. Это бывает, когда часть строки обеспечивает уникальность выбора значения. Например:
```Gherkin
И из выпадающего списка с именем 'Контрагент' я выбираю по строке "ООО Ромаш"
И я перехожу к следующему реквизиту
Тогда элемент формы с именем 'Контрагент' стал равен "ООО Ромашка"
```
В этом случае текст в шаге выбора по строке и в шаге проверяющего значения может отличаться.
Важный момент: при генерации шагов, которые используют ввод данных в поле по строке нужно стараться использовать строку целиком. В приведенном выше примере это будет "ООО Ромашка", а не "ООО Ромаш".

## 2. Для ссылочных полей с типом Перечисление это будет выглядеть так:
```Gherkin
И из выпадающего списка с именем 'ТипСклада' я выбираю точное значение "Основной"
И я перехожу к следующему реквизиту
Тогда элемент формы с именем 'ТипСклада' стал равен "Основной"
```
Т.е. какое значение было выбрано в поле, то оно же используется при проверке значения поля

## 3. Для числовых полей это будет выглядеть так:
```Gherkin
И в таблице "Товары" в поле с именем 'ТоварыЦена' я ввожу текст "100"
И в таблице "Товары" я завершаю редактирование строки
Тогда в таблице "Товары" поле с именем 'ТоварыЦена' имеет значение "100,00"
```
В данном примере видно, что значение вводилось без копеек, но проверять надо полное значение поля, т.к. для числовых полей это очень важно.

Вопрос: Мне нужно провести документ. Для этого есть кнопки "Провести" и "ПровестиИЗакрыть". Как сделать лучше?
Ответ: Важно понимать, что проведение документов может занимать какое-то время.
Если после проведения документ форма документа больше не нужна, то лучше использовать кнопку "ПровестиИЗакрыть". Пример:
```Gherkin
И я нажимаю на кнопку с именем 'ФормаПровестиИЗакрыть'
И я жду закрытия окна "ЗаголовокОкнаДокумента" в течение 20 секунд
```
Это таким образом получится гарантированно дождаться окончания проведения документа, т.к. это произойдет после того как окно закрылось, а шаг
```Gherkin
И я жду закрытия окна "ЗаголовокОкнаДокумента" в течение 20 секунд
```
позволит этого корректно дождаться.

Если после проведения документа планируется работать с его формой дальше, то лучше сделать так.
* Если документ новый и ещё ни разу не записывался в базу, то нажать на кнопку записи, дождаться когда заполнится поле Номер, потом сохранить навигационную ссылку на документ в переменную, нажать на кнопку "ПровестиИЗакрыть", дожаться что форма документа закроется, затем открыть форму документа по ссылке. Вот шаги для этого:
```Gherkin
И я нажимаю на кнопку с именем 'ФормаЗаписать'
И я жду, что поле с именем 'Номер' перестанет быть пустым в течение 30 секунд
И я сохраняю навигационную ссылку текущего окна в переменную "ПеременнаяГдеХранитсяСсылкаНаДокумент"
И я нажимаю на кнопку с именем 'ФормаПровестиИЗакрыть'
И я жду закрытия окна "ЗаголовокОкнаДокумента" в течение 20 секунд
Дано Я открываю навигационную ссылку "$ПеременнаяГдеХранитсяСсылкаНаДокумент$"
```

* Если документ не новый и уже есть в базе данных, то надо сохранить навигационную ссылку на документ в переменную, нажать на кнопку "ПровестиИЗакрыть", дожаться что форма документа закроется, затем открыть форму документа по ссылке. Вот шаги для этого:
```Gherkin
И я сохраняю навигационную ссылку текущего окна в переменную "ПеременнаяГдеХранитсяСсылкаНаДокумент"
И я нажимаю на кнопку с именем 'ФормаПровестиИЗакрыть'
И я жду закрытия окна "ЗаголовокОкнаДокумента" в течение 20 секунд
Дано Я открываю навигационную ссылку "$ПеременнаяГдеХранитсяСсылкаНаДокумент$"
```

Важный момент: если заголовок окна имеет вид "Поступление товаров и услуг 000000002 от 10.05.2026 0:51:27 *", то параметр ЗаголовокОкнаДокумента должен быть равен "Поступление товаров и услуг 000000002 от *".
Чтобы исключить проблему, что документ может создаваться текущей датой и временем и тогда заголовок окна при каждом запуске сценария будет разный.
А использование символа * позволит исключить из проверки дату и время документа.

Вопрос: Как лучше использовать инструмент run_scenario: с mode=reloadAndRun или mode=reloadAndRunFromLine?
Ответ: Лучше использовать reloadAndRunFromLine, если это возможно. Так цикл отладки и разработки сценария тестирования сократится.
Особенно выгодно использовать reloadAndRunFromLine если в при выполнении сценария упал шаг и этот шаг был исправлен или доработан.
Тогда в параметре lineNumber надо передать номер строки feature файла, в которой находится этот шаг.

# Когда использовать reloadAndRunFromLine:

## Рекомендуется использовать reloadAndRunFromLine в следующих случаях:

1. **После исправления ошибки в середине/конце сценария**
   - Если начальные шаги уже работают стабильно
   - Если исправлен шаг с номером строки > 20 (когда сценарий уже достаточно длинный)
   - Пример: после 5-го запуска сценария, когда шаги 1-16 уже работают

2. **При разработке длинных сценариев (более 50 шагов)**
   - Не тратить время на повторение уже отлаженных частей
   - Экономия времени на каждом цикле отладки

3. **Когда исправление не влияет на предыдущие шаги**
   - Если изменили шаг в разделе "Заполнение полей" — запускать с этого шага
   - Если изменили шаг в разделе "Проверка отчета" — запускать с шага открытия отчета

## Когда лучше использовать reloadAndRun:

1. **Первый запуск нового сценария** — всегда с начала
2. **Когда исправление может повлиять на предыдущие шаги** — например, изменили имя переменной
3. **Финальная проверка** — перед завершением работы убедиться, что всё работает с начала до конца

## Практическое правило 3-х запусков:

1. Первый запуск — всегда `reloadAndRun` (проверка с начала)
2. Второй и последующие запуски при отладке — `reloadAndRunFromLine` с номера строки упавшего шага
3. Финальный запуск — снова `reloadAndRun` (подтверждение полной работоспособности)

## Пример использования:

Допустим, сценарий упал на шаге 45 из 60. Вы исправили шаг 45.
Вместо:
```
mode: "reloadAndRun"  // Выполнит все 60 шагов
```

Используйте:
```
mode: "reloadAndRunFromLine"
lineNumber: 45  // Выполнит шаги с 45 по 60
```

Это сократит время выполнения сценария в 2-3 раза на каждом цикле отладки.

## Важное замечание:

При использовании reloadAndRunFromLine:
- Убедитесь, что контекст формы/окна соответствует ожидаемому на момент запуска с указанной строки
- Если сценарий требует определённого состояния системы (например, открытый документ), убедитесь, что это состояние уже достигнуто
- Периодически (после нескольких итераций) запускайте сценарий полностью с начала для финальной проверки

Вопрос: Я пишу сценарий, который создаёт новый документ. И во время написания и отладки сценария я запускаю его несколько раз и тестовый документ создаётся тоже несколько раз, что мешает следующему запуску сценария.
Ответ: Да, действительно. Может возникнуть ситуация, когда во время написания сценария его надо запустить несколько раз и тестовый документ, например, "ПоступлениеТоваров" будет создан несколько раз.
И из-за этого возникнут проблемы при проверке остатков товара на складе, т.к. остатки будут увеличиваться при создании каждого нового документа.
Чтобы решить проблему нужно следовать следующим правилам:

# Правило служебного комментария
При создании нового документа в поле Комментарий нужно вводить специальный текст, который состоит из двух частей: строки "ДокументСозданВСценарииАвтотестирования" и уникальной части, например "ДокументСозданВСценарииАвтотестирования_1" таким шагом:
```Gherkin
И в поле с именем 'Комментарий' я ввожу текст "ДокументСозданВСценарииАвтотестирования_1"
```
Далее в сценарии эту уникальную строку можно будет использовать, чтобы найти нужный документ. Например задать отбор в форме списка по этой уникальной строке.

# Правило пометки удаления
Нужно в начале сценария создать специальный блок, который будет помечать на удаление документы, которые были созданы предыдущими запусками сценария.

## Если сценарий создает один вид документов:

Этот блок должен делать следующее:
1. Открыть список документов
2. Сделать отбор по колонке комментарий: Комментарий="ДокументСозданВСценарииАвтотестирования". Этот отбор сработает корректно для всех строк, которые содержат строку "ДокументСозданВСценарииАвтотестирования", например для строки "ДокументСозданВСценарииАвтотестирования_1".
3. Выделить все строки.
4. Вызвать команду пометки на удаление.
Вот шаги, которые это выполняют:
```Gherkin
* Пометка на удаление документов, созданных предыдущими запусками
    Дано В командном интерфейсе я выбираю "Закупки" "Документы закупки (все)"
    Тогда открылось окно "Документы закупки (все)"
    И в таблице 'СписокДокументыЗакупки' я активизирую поле с именем 'СписокДокументыЗакупкиКомментарий'
    И я выбираю пункт контекстного меню с именем 'СписокНайти1' на элементе формы с именем 'СписокДокументыЗакупки'
    Тогда открылось окно "Найти"
    И в поле с именем 'Pattern' я ввожу текст "ДокументСозданВСценарииАвтотестирования"
    И я нажимаю на кнопку с именем 'Find'
    Тогда открылось окно "Документы закупки (все)"
    И в таблице 'СписокДокументыЗакупки' я выделяю все строки
    И я выбираю пункт контекстного меню с именем 'СписокКонтекстноеМенюУстановитьПометкуУдаления' на элементе формы с именем 'СписокДокументыЗакупки'
    Тогда появилось окно предупреждения системы
    И я запоминаю текст предупреждения в переменную "ТекстМодальногоДиалога"
    Если переменная "ТекстМодальногоДиалога" имеет значение "Пометить выделенные элементы на удаление?" Тогда
        И я нажимаю на кнопку 'Да'
    ИначеЕсли переменная "ТекстМодальногоДиалога" имеет значение "Снять с выделенных элементов пометку на удаление?" Тогда
        И я нажимаю на кнопку 'Да'
        И я выбираю пункт контекстного меню с именем 'СписокКонтекстноеМенюУстановитьПометкуУдаления' на элементе формы с именем 'СписокДокументыЗакупки'
        Тогда появилось окно предупреждения системы
        И я нажимаю на кнопку 'Да'
    Иначе
        Тогда я закрываю окно предупреждения
```

## Если сценарий создает несколько видов документов:

Если сценарий создает несколько видов документов (например, документы закупки и заказы на перемещение), нужно создать **отдельные блоки удаления для каждого вида документов**:

```Gherkin
* Пометка на удаление документов, созданных предыдущими запусками
    * Удаление документов закупки
        Дано В командном интерфейсе я выбираю "Закупки" "Документы закупки (все)"
        ... (поиск по комментарию и пометка на удаление)
    
    * Удаление заказов на перемещение
        Дано Я открываю навигационную ссылку "e1cib/command/Документ.ЗаказНаПеремещение.ОткрытьСписок"
        ... (поиск по комментарию и пометка на удаление)
```

**Важно:**
- Блок очистки должен быть в начале сценария, перед основной логикой
- Для каждого вида документов — отдельный блок удаления с уникальным поиском

Вопрос: Я использовал инструмент manage_command_interface с параметром action=click_function_panel. Долго ли будет открываться соответствующее окно, после выполнения команды?
Ответ: Если выполнение команды в панели функций выполняется в сеансе клиента тестирования впервые, то команда в первый раз может выполняться долго. При повторных выполнения платформа 1С:Предприятие кеширует служебные данные и команда будет выполняться быстрее.

Вопрос: У меня открыта форма элемента справочника или форма элемента документа. Я хочу понять какие данные есть базе данных клиента тестирования, чтобы понять какие значения я могу вводить в поля с типом Справочник, Документ, Перечисление.
Ответ: Для этого надо вызвать инструмент get_table_data. Он позволяет посмотреть какие записи есть в таблицах базы данных.
Например, если у поля тип это СправочникСсылка.Организации, то можно получить данные из справочника организации и так далее.

Вопрос: У меня открыта форма элемента справочника или форма элемента документа. Я хочу понять какие типы данных у полей формы. Как это сделать?
Ответ: Для этого надо вызвать инструмент get_object_attributes. Он возвращает значения полей объекта, связанного с формой. При этом для каждого поля указывается его тип, например, СправочникСсылка.Контрагенты.

Вопрос: Как понять данные из каких таблиц тестовый базы нужно прочитать с помощью инструмента get_object_attributes?
Ответ: Если в задаче явно не указаны какие именно тестовые данные нужно использовать в сценарии, то нужно сделать так:
- когда в клиенте тестирования открыта форма документа или справочника, которые надо создать в сценарии, надо вызвать инструмент get_object_attributes и понять какие типы полей используются в форме документа или справочника.
- потом нужно для ссылочного типа данных надо вызвать инструмент get_table_data, чтобы понять какие тестовые данные есть в конкретной таблице тестовой базы данных

Вопрос: Мне нужно в сценарии сделать проверку отчета. Что это значит?
Ответ: Это значит, что нужно открыть форму отчета, сформировать его и сделать сравнение табличного документа, который формирует отчет, с эталонным значением.
Если в условиях решаемой задачи нет заранее созданного эталона, то это значит, что эталон надо будет сохранить на диск с помощью инструмента save_table_document_to_file.
Рекомендуется сохранять эталон рядом с feature файлом ("рядом" означает в том же каталоге или подкаталоге). Формат файла - mxl.

Вопрос: Мне нужно написать проверку, которая сравнивает табличный документ с эталоном. Как это сделать?
Ответ: Для этого есть специальные шаги.
# Если нужно сделать строгое сравнение табличного документа с эталоном, то нужно использовать:
```Gherkin
Дано Табличный документ "ИмяРеквизита" равен макету "ИмяМакета"
```
где ИмяРеквизита - это имя элемента формы, в котором находится табличный документа, а ИмяМакета - это имя файла эталона без расширения.
Рекомендуется хранить эталоны табличных документов рядом с feature файлом.
Например, если рядом с feature файлом лежит файл ОстаткиТоваров.mxl, то проверка будет выглядеть так:
```Gherkin
Дано Табличный документ "ИмяРеквизита" равен макету "ОстаткиТоваров"
```
Важный момент, файл эталона может также находится и в подкаталогах относительно каталога, в котором находится feature файл.
Например, если используется файл c:\features\Поступления\ПоступлениеТоваров.feature, то файл эталона может находится в каталоге c:\features\Поступления\Файлы\ОстаткиТоваров.mxl
При этом текст шага будет таким же:
```Gherkin
Дано Табличный документ "ИмяРеквизита" равен макету "ОстаткиТоваров"
```
т.к. Vanessa Automation будет искать файлы в подкаталогах тоже.

Важный момент. Во всех шагах, сравнивающих табличный документ с эталоном, если текст ячейки эталона будет равен символу *, то это значит, что данная ячейка проверяться не будет.

# Если нужно сделать частичное сравнение табличного документа с эталоном, то нужно использовать:
```Gherkin
Дано Табличный документ "ИмяРеквизита" равен макету "ОстаткиТоваров" по шаблону
```
Данный шаг позволяет делать частичное сравнение ячеек табличного документа с эталоном. Если в эталоне в тексте ячейки часть строки заменена на символ *, то это значит что вместо * может быть произвольная строка.
Например в эталоне в ячейке находится текст "Документ №10 от *", а в табличном документе в клиенте тестирования в той же ячейке находится "Документ №10 от 01.02.2026", то при использовании шага
```Gherkin
Дано Табличный документ "ИмяРеквизита" равен макету "ОстаткиТоваров" по шаблону
```
эти ячейки будут считаться равными.

Вопрос: Как прочитать содержимое табличного документа?
Ответ: Основной способ — использовать инструмент `get_form_element_data` для получения данных табличного документа:
1. Убедиться, что сейчас открыто окно, в котором находится нужный табличный документ, например, окно отчета
2. Вызвать инструмент `get_form_element_data` с параметром `element_name` — имя элемента формы (табличного документа) для получения данных.
Инструмент возвращает информацию об элементе в формате Markdown: имя, заголовок, тип, вид, значение, представление, флаги (Видимость, Доступность, ТолькоПросмотр).

## Запасной способ: сохранение в файл
Если нужно сохранить содержимое табличного документа в файл, используйте инструмент `save_table_document_to_file`:
1. Убедиться, что сейчас открыто окно, в котором находится нужный табличный документ
2. Вызвать инструмент `save_table_document_to_file` с параметрами:
   - `file_name` — полный путь к файлу для сохранения
   - `form_element_name` — имя элемента формы (табличного документа)
   - `format` — формат файла: 'mxl' (по умолчанию), 'xlsx' (Excel), 'pdf' (PDF)

## Важное замечание: неправильно для получения значения табличного документа использовать шаг:
```Gherkin
И я сохраняю данные элемента формы с именем "ОтчетТабличныйДокумент" в файл
```
Этот шаг используется только в редких случаях, когда в сценарии пользователя файлы нужно сохранить на диск.
Если нужно выполнить проверку содержимого табличного документа с эталоном, то про это рассказано в вопросе: 'Мне нужно написать проверку, которая сравнивает табличный документ с эталоном. Как это сделать?'

Вопрос: Как найти элементы формы (кнопки, поля, таблицы) при неизвестном имени в клиенте тестирования?
Ответ: Используйте инструмент `get_form_analysis` с различными параметрами для получения информации о форме:

**Параметры инструмента:**
- `format=elements` - получить все элементы формы с их именами и заголовками
- `format=buttons` - получить только кнопки формы
- `format=gherkin` - получить шаги Gherkin для работы с формой

**Пример использования:**
```
инструмент: get_form_analysis
параметр: format=buttons
```

**Результат возвращает:**
- Имя кнопки и заголовок кнопки
- Имя поля и заголовок поля
- Имя таблицы и её структуру

**Алгоритм поиска:**
1. Вызвать `get_form_analysis` с параметром `format=buttons` для поиска кнопок
2. Найти нужную кнопку по заголовку в результате
3. Использовать имя кнопки в сценарии: `И я нажимаю на кнопку с именем 'ИмяКнопки'`

Вопрос: Как правильно исправлять ошибки при выполнении сценариев в клиенте тестирования?
Ответ: Используйте следующий алгоритм отладки:

**Алгоритм исправления ошибок:**
1. **Сделать скриншот окна** через `get_window_screenshot_os`:
   ```
   инструмент: get_window_screenshot_os
   параметр: window_title="Заголовок окна"
   параметр: color_mode="grayscale"
   ```

2. **Получить анализ формы** через `get_form_analysis`:
   ```
   инструмент: get_form_analysis
   параметр: format=elements
   ```

3. **Получить список доступных кнопок/полей** из результата анализа формы

4. **Исправить имя элемента в сценарии** на основе полученных данных

5. **Запустить сценарий с `mode=reloadAndRunFromLine`** для ускорения отладки:
   ```
   инструмент: run_scenario
   параметр: mode="reloadAndRunFromLine"
   параметр: lineNumber=<номер строки упавшего шага>
   ```

**Важно:** Использование скриншотов помогает визуально понять текущее состояние формы и найти нужные элементы.

Вопрос: Как правильно использовать локальные переменные при сохранении навигационной ссылки?
Ответ: При сохранении навигационной ссылки в переменную **НЕЛЬЗЯ** использовать символы `$` в имени локальной переменной.

**Правильно:**
```Gherkin
И я сохраняю навигационную ссылку текущего окна в переменную "НавигационнаяСсылка"
```

**Неправильно:**
```Gherkin
И я сохраняю навигационную ссылку текущего окна в переменную "$НавигационнаяСсылка$"
```

**При использовании переменной в сценарии** обрамлять символами `$`:
```Gherkin
Дано Я открываю навигационную ссылку "$НавигационнаяСсылка$"
```

**Важно:** Шаг `И я сохраняю навигационную ссылку текущего окна в переменную` принимает имя переменной БЕЗ символов `$`. Символы `$` используются только при чтении переменной в других шагах.

Вопрос: Какие особенности именования окон в клиента тестирования следует учитывать при написании тестов?
Ответ:

**Особенности именования окон:**

1. **Окна списков документов** имеют заголовок во множественном числе:
   - ✅ "Заказы на перемещение товаров"
   - ❌ "Заказ на перемещение"

2. **Окна создания документов** имеют суффикс "(создание)*":
   - "Заказ на перемещение товаров (создание)*"
   - "Приобретение товаров и услуг (создание)*"
