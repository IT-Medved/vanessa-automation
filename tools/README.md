# Утилиты для контрибьютора (Windows)

Папка `tools/` содержит утилиты для разработки и тестирования Vanessa Automation:

- сборка `.epf` из исходников в XML-формате;
- запуск behavior-тестов (Gherkin-сценариев) локально на нескольких версиях 1С:Предприятия;
- формирование и просмотр отчётов Allure о прохождении тестов.

Все скрипты написаны на OneScript и cmd, запускаются локально под Windows. Linux-аналоги — в [`tools/linux/`](./linux/).

## Кратко

Если всё уже установлено и настроено (см. [Что нужно установить](#что-нужно-установить)) — ежедневный цикл из корня репозитория:

```cmd
Compile.bat
cd tools
"1 PrepareCheck.cmd"
"MiddleCheck_8327_UF.cmd"
"3 ViewAllureReport.cmd"
```

- `Compile.bat` собирает `.epf` (около 8 минут на чистом репозитории, секунды на повторных запусках).
- `1 PrepareCheck.cmd` создаёт сервисные базы (один раз).
- `MiddleCheck_8327_UF.cmd` прогоняет тесты на 8.3.27 в режиме управляемых форм.
- `3 ViewAllureReport.cmd` открывает HTML-отчёт Allure в браузере.

## Что нужно установить

### Программное обеспечение

| Инструмент | Версия | Назначение |
|---|---|---|
| 1С:Предприятие | **8.3.17** или новее для `Compile.bat`; конкретная версия (например 8.3.27) для прогона тестов | runtime для сборки `.epf` через Конфигуратор и для запуска Gherkin-сценариев |
| OneScript | **v2.0.1** (latest, рекомендуется) или **v1.9.4** (LTS, ветка 1.x) — https://github.com/EvilBeaver/OneScript/releases | исполняет скрипты `*.os` в `tools/onescript/` (сборка, подготовка баз, запуск сессий) |
| Java | **21 LTS** или новее | требуется для работы Allure |
| Allure | **2.x** — https://github.com/allure-framework/allure2/releases | формирует HTML-отчёт из XML-результатов прогона |

После установки команды `oscript` и `allure` должны быть доступны в `PATH`.

**Сколько версий 1С нужно ставить.** Для локальной разработки достаточно одной (например 8.3.27) — её хватит на любой `*_8327_*`-скрипт. Полный набор из примерно 12 версий, перечисленных в `tools/JSON/Main.json`, нужен только при подготовке релиза или для прогона `2 CheckBehavior.cmd` целиком. Если запрошенной в скрипте версии нет, библиотека `v8runner` подставит самую старшую установленную без предупреждения (см. [Решение проблем](#решение-проблем)).

### Конфигурация 1С

После установки 1С нужны два обязательных шага настройки.

**Путь установки.** По умолчанию 1С ставится в `C:\Program Files\1cv8\<версия>\` — все JSON-конфиги в `tools/JSON/VBParams*.json` рассчитаны именно на этот путь. Если 1С уже установлена в другом месте (на другом диске, в `C:\Program Files (x86)\1cv8\` для 32-битной установки и т.п.), править сотню JSON-конфигов нерационально. Простейшее решение — junction-link от ожидаемого пути к реальному. В cmd от имени администратора:

```cmd
mklink /J "C:\Program Files\1cv8" "D:\путь-к-реальной-установке-1cv8"
```

После этого `dir "C:\Program Files\1cv8"` покажет подпапки версий (например `8.3.27.1936`).

**Отключение защиты от опасных действий.** Начиная с 1С:Предприятие 8.3.9.2033 платформа при каждом открытии внешней обработки показывает модальный диалог «Предупреждение безопасности». Тесты VA загружают `init.epf`, `vanessa-automation.epf` и десятки служебных `.epf` — без отключения защиты прогон останавливается на первом же диалоге.

Добавьте строку `DisableUnsafeActionProtection=.*` в файл конфигурации платформы `conf.cfg`. Файл лежит в двух местах:

- `C:\Program Files\1cv8\conf\conf.cfg` — общий для всех версий;
- `C:\Program Files\1cv8\<версия>\bin\conf\conf.cfg` — версионный.

Добавьте строку в оба файла для надёжности. Подробнее — `docs/FAQ/index.md`, пункт 15.

## Порядок работы

### Сборка `.epf` (`Compile.bat`)

В репозитории лежат исходники `VanessaAutomation.epf` в XML-формате; готового `.epf` в репозитории нет. `Compile.bat` в корне репозитория собирает его за три этапа:

1. **Скачивание внешних зависимостей** из GitHub Releases — VanessaExt (бинарная компонента: скриншоты, видео, горячие клавиши) и VAEditor (редактор Gherkin). Версии зафиксированы в `lib/packages.json`. Если файлы существуют и хеш совпадает, скачивание пропускается.
2. **Упаковка шаблонов** — JS-библиотеки из `lib/VA_Chrome/` и Gherkin-ключевые слова из `lib/Keywords/` собираются в ZIP-архивы `Template.bin`.
3. **Сборка `.epf`** — Конфигуратор пересобирает 208 обработок и отчётов из XML-исходников. На повторных запусках проверяются `fileversion`-метки, и неизменённые исходники пропускаются.

Запуск из корня репозитория:

```cmd
Compile.bat
```

Результат — файл `vanessa-automation.epf` в корне. Чистая сборка занимает около 8 минут, инкрементальная — секунды.

Запускать перед первым прогоном тестов и после каждого изменения исходников VA. Скрипты `MiddleCheck_*.cmd` и `FastCheck_*.cmd` сами вызывают `compile.bat` перед прогоном, отдельно его перезапускать не нужно.

### Подготовка ИБ (`1 PrepareCheck.cmd`)

Для каждой версии платформы 1С тесты используют отдельную информационную базу в `tools/ServiceBases/`. Скрипт `1 PrepareCheck.cmd` создаёт их по одной на каждую версию из встроенного списка (около 9 баз: 8.3.6, 8.3.21–25, 8.3.27, 8.5.1, плюс NoSync-вариант).

```cmd
cd tools
"1 PrepareCheck.cmd"
```

Первый запуск занимает несколько минут: создаются файловые ИБ, в каждую загружается тестовая конфигурация из `lib/CF/83/`, обновляется `Configuration.xml` с правильным `CompatibilityMode`. На повторных запусках уже существующие базы не пересоздаются.

Запускать один раз после `Compile.bat`, а также при добавлении новой версии 1С или после удаления `tools/ServiceBases/`.

> Все cmd-скрипты в `tools/` запускаются из папки `tools/`, не из корня репозитория. Внутри них стоит относительный `cd .\..\` или `cd ..` — попытка запустить через `call "tools\1 PrepareCheck.cmd"` из корня уведёт скрипт на уровень выше корня, и git выдаст ошибку «not a git repository».

### Запуск тестов

Три способа от самого быстрого к самому полному:

```cmd
cd tools
"FastCheck_<версия>_<режим>.cmd"        :: быстрая
"MiddleCheck_<версия>_<режим>.cmd"      :: средняя
"2 CheckBehavior.cmd"                   :: полная (для всех платформ)
```

Соотношение объёмов — см. [Состав FastCheck / MiddleCheck / CheckBehavior](#состав-fastcheck--middlecheck--checkbehavior). Готовые имена скриптов под версии — см. [Готовые скрипты в репозитории](#готовые-скрипты-в-репозитории).

**Что выбирать.** В процессе разработки используется `FastCheck_<своя-версия>_*.cmd` (если он существует в репозитории для нужной версии — иначе `MiddleCheck_<версия>_*.cmd`). Перед отправкой PR обязательно прогоняется `MiddleCheck` на своей версии. `2 CheckBehavior.cmd` запускают только в CI или перед релизом.

### Просмотр отчёта (`3 ViewAllureReport.cmd`)

После прогона в `tools/ServiceBases/allurereport/` лежат XML-результаты по каждому сценарию. Скрипт собирает из них HTML-отчёт через `allure generate` и открывает в браузере:

```cmd
cd tools
"3 ViewAllureReport.cmd"
```

Отчёт показывает прошедшие и упавшие сценарии, шаги, тайминги, скриншоты в точках падения.

## Справочник

### Состав FastCheck / MiddleCheck / CheckBehavior

| Вариант | Скрипт | Фич в наборе\* | Что прогоняется |
|---|---|---|---|
| Быстрая проверка | `FastCheck_<версия>_*.cmd` | 378 | весь набор фич за вычетом помеченных `@IgnoreOnFastCheck`, `@Video` и тегами под чужие платформы и режимы (`@IgnoreOn82Builds`, `@IgnoreOnOFBuilds` и т.п.) |
| Средняя проверка | `MiddleCheck_<версия>_*.cmd` | 384 (две части: 223 + 163) | надмножество FastCheck — добавляет 6 фич (5 с `@Video` и 1 с `@IgnoreOnFastCheck`). Внутри MiddleCheck фичи разбиты тегами `@uf-part1` и `@uf-part2` и прогоняются двумя последовательными частями для параллелизации в CI |
| Полная проверка | `2 CheckBehavior.cmd` | 2 773 прогона / 387 уникальных фич | 12 сборок из `tools/JSON/Main.json` — те же фичи прогоняются на разных платформах (8.2, 8.3.6, 8.3.17–8.3.27 UF, 8.3.27 Web) и в разных режимах (обычные и управляемые формы, совместимость с 8.2, без синхронизации). Самые крупные сборки — UF на 8.3.21–27 (около 380 фич каждая); самые маленькие — 8.2 OF и 8.3.6 OF (около 44) |

\* Всего в репозитории 1163 `.feature`-файла. Самостоятельными тестами прогоняются 387 из них — те, что входят хотя бы в одну сборку из `tools/JSON/Main.json`. Остальные 776 как тесты не запускаются: `Support/UF9_Вспомогательные_фичи` (около 630 файлов) подключаются как Gherkin-шаги из основных фич, то есть служат библиотекой шагов; `Drafts/` и `SikuliXServer/` — черновики и устаревшее; `StoryMapper/` — материалы карты историй. Часть таких фич дополнительно помечена тегом `@IgnoreOnCIMainBuild`. Числа в таблице приведены для одной из самых полных конфигураций (UF на современной 1С); на старых платформах и в специфичных режимах применимых фич меньше.

### Суффиксы в именах cmd-скриптов

- `_UF` — управляемые формы (современный режим 1С);
- `_Server` — серверная база (клиент-серверная, не файловая);
- `_nosync` — без синхронизации данных между базами.

Примеры:

- `MiddleCheck_8327_UF.cmd` — средняя проверка на 8.3.27, управляемые формы;
- `MiddleCheck_8323_Server_UF.cmd` — средняя проверка на 8.3.23, клиент-серверный режим, управляемые формы;
- `FastCheck_8322_UF_nosync.cmd` — быстрая проверка на 8.3.22, управляемые формы, без синхронизации.

### Готовые скрипты в репозитории

В `tools/` лежат `FastCheck_*.cmd` только для версий 1С: **8.3.17**, **8.3.19**, **8.3.22**. Все они в JSON-конфиге указывают `КаталогПоискаВерсииПлатформы: C:\Program Files (x86)\1cv8` (32-битная установка). Если установлена только 64-битная 8.3.27, ни один из существующих FastCheck напрямую не запустится — используйте `MiddleCheck_8327_UF.cmd`.

Готовые `MiddleCheck_*.cmd` есть для версий 8.3.17, 8.3.19, 8.3.23, 8.3.25, 8.3.27 (UF и Server-варианты), 8.5.1.

## Решение проблем

**Скрипт падает с ошибкой `fatal: --local can only be used inside a git repository`.**
Скрипт запущен не из папки `tools/`. Внутри cmd-файлов стоит `cd .\..\` или `cd ..` — они рассчитаны на запуск именно из `tools/`. Корректный запуск: `cd <repo>\tools` и затем `"<имя-скрипта>.cmd"`.

**1С запустилась, но остановилась на модальном окне «Предупреждение безопасности».**
Не отключена защита от опасных действий. См. [Конфигурация 1С](#конфигурация-1с) — нужно добавить `DisableUnsafeActionProtection=.*` в `conf.cfg`. После правки закройте текущую 1С (`Alt+F4` или «Нет» в диалоге) и запустите прогон заново — настройки `conf.cfg` читаются на старте процесса, текущей сессии правка не поможет.

**При первом прогоне на свежей сервисной базе появляется окно «Внешняя компонента успешно установлена».**
Это не ошибка. VanessaExt (бинарная компонента: скриншоты, видео, горячие клавиши) регистрируется в инфобазе. Нажмите «OK», прогон продолжится. При следующих запусках на этой же базе окно не появится.

**Файлы `tools/ServiceBases/log<версия>.txt` и `BuildStatus.log` не создаются после запуска.**
Тест не дошёл до выполнения сценариев — остановился на этапе инициализации. Самые частые причины:

- не отключена защита от опасных действий (см. выше) — 1С остановилась на модальном диалоге;
- 1С не нашла нужную версию платформы по `КаталогПоискаВерсииПлатформы` из VBParams-конфига — проверьте, что путь в JSON соответствует реальному расположению.

**`1 PrepareCheck.cmd` создал базы для версий, которые не установлены.**
Это штатное поведение `v8runner.ПолучитьПутьКВерсииПлатформы`: если запрошенная версия не найдена, библиотека подставляет самую старшую установленную. Базы создаются, но через единственный доступный движок (например, на машине только с 8.3.27 база `v83ServiceBase8321` тоже создастся, но загрузка пройдёт через 8.3.27). Это не блокирует работу: `MiddleCheck_<версия>_*.cmd` сами стартуют именно ту платформу, что в их имени, — если её нет, упадёт только этот конкретный прогон, остальные останутся рабочими.

**Куда смотреть логи прогона.** Помимо stdout cmd-скрипта, во время прогона в `tools/ServiceBases/` появляются:

- `log<версия>.txt` — текстовый лог выполнения сценариев (шаги, тайминги);
- `Messages <версия> UF.txt` — сообщения 1С через `Сообщить()`;
- `BuildStatus.log` — финальный статус прогона (сколько прошло и упало);
- `ScreenShots/` — скриншоты экрана в точках падения;
- `allurereport/`, `cucumber/`, `junitreport/` — машинные результаты в трёх форматах.
